01. What is one “fact” we store?

Every number in the system is stored as a single, granular observation to prevent estimates or double-counting:

Field Meaning Example from your world
When Business date the inspection/production happened 2025-04-02
Stage Process / inspection gate Visual Inspection, Balloon Testing, Valve Integrity, Final
Size French size (when sheet is size-wise) 16 FR
Batch Work unit / lot identifier (when present) 25A28 (seen on VISUAL-style sheets)
Measure What was counted Checked / Accepted / Hold-Rework / Rejected
Defect Rejection reason (when broken down) COAG, PINH, STBL, LEAK, OTH, …
Qty Integer count (pcs; sometimes trolleys) 120
Source Where the number came from File name + sheet + cell, or manual entry
System Rule: We do not invent or interpolate numbers. Missing data fields show as blank/empty, never estimated or filled by default.

02. Dimensions & Batch Definition

Assumptions establishing what "batch" and process tracking variables represent:

ID Assumption details Confirm / Correct
[A1]
A batch is a lot / batch number written on the inspection sheet (e.g. 25A28), not a trolley and not “the whole day.”
[A2]
One batch can appear on more than one stage (e.g. same lot later at Balloon / Final) — stages are separate facts, not one merged row.
[A3]
Batch is optional. Many rows in your books are day-only (no batch column). Those stay valid with batch = (none).
[A4]
Size and batch are independent: same day can have multiple sizes and multiple batches.
[A5]
Shift (Day/Night) is metadata for entry/audit, not the primary KPI axis (unless you say otherwise).
[A6]
Machine / operator / supervisor are optional tags, not required for rejection rate math.

03. Aggregation & Math Rules

Decisions that establish how metrics roll up and how data discrepancies are handled:

ID Situation details & Default rule Confirm / Correct
[A7]
Same stage + day + size, different batches: Keep as separate batch rows. For daily/monthly KPIs, sum them.
[A8]
Same stage + day, one file has per-size rows and also a whole-line total: Prefer size rows (sum = stage total); drop the redundant whole-line total to avoid double count.
[A9]
Same stage + day described by two different uploaded files: Keep the higher-precedence source only (size-wise / detailed > summary). Direct manual entry wins over upload for that day.
[A10]
Re-upload of the same file: **Idempotent** — does not double-count.
[A11]
Sheet has a % rejection formula cell: Store as a claim to verify, not as the source of truth. Dashboard recomputes rate from checked & rejected.
[A12]
Defect columns sum ≠ Rejected total: Flag as a data issue (warning); still store both; do not silently “fix” the sheet.
[A13] When a manager opens “Batch view,” should they see unmerged batch lines (even if same day/size)? (Default: YES)
[A14] When a manager opens “Monthly dashboard,” should they see sums across batches for that month? (Default: YES)

06. Source of Truth (Conflict Resolution)

Rules that determine what wins when data entries disagree:

ID Conflict Situation & Resolution Rule Confirm / Correct
[A19]
Excel upload vs manual data entry for same stage·day: Manual entry wins.
[A20]
Detailed size-wise book vs monthly summary book: Detailed book wins.
[A21]
Two “official” numbers from different departments: System flags conflict; human adjudicates — we do not pick silently.
[A22]
Column header spelling variants (“STRUCK BALLOON” / “BALLOOM BRUST”): Map to one defect code after you verify mappings once; keep raw text for audit.

07. MLP Scope & Priority Ranking

Features slated for the initial Minimum Viable Product (MLP) launch versus later updates:

In Scope (v1 Standard)

  1. Workbook uploads with column mapping (numbers match sheet perfectly).
  2. Day and Batch views (where batch numbers are captured).
  3. Stage, size, and defect breakdowns with dynamic "View Source" provenance.
  4. Zero silent double counting (idempotent uploads).
  5. Export & audit system showing raw entries.

Explicitly Later (Post-v1)

  • Full SPC suite statistical polish.
  • COPQ finance-grade cost structures.
  • CAPA ticket automation.
  • Multi-plant / multi-product data mapping.
  • AI natural language chat interface.
