1. The Core Judgment Call

The PRD defines 8 complete analytics engines and a fully functional Lean manufacturing suite. However, a pharma SME will form its defining opinion of MO!D in the first 15 minutes of use, based entirely on four specific experiences:

1. File Understanding EXPERIENCE #1

"I uploaded our files and it understood them."

Seamless, robust spreadsheet ingestion and alias mapping coupled with interactive data verification visualizers.

2. Personalized Views EXPERIENCE #2

"It shows ME what I care about."

Dashboards personalized by role, displaying tailored information density and context for shop-floor operators up to executive owners.

3. High-Trust Calculations EXPERIENCE #3

"I can trust these numbers."

Transparent data provenance. Every chart, metric, and finding points back to its exact origin cell or logical rule through interactive visual "beams".

4. Hard Rupee Metrics EXPERIENCE #4

"It tells me where I'm losing money."

Converting plant defects and rejection percentages directly into Pareto charts of hard financial losses (COPQ in ₹) to drive immediate executive decisions.

Directive: Build Adoption Value, Lock Expansion Value
Build the four experiences above to a state of absolute, flawless polish. Represent all advanced capabilities (SPC, fishbones, 5-Why root cause, FMEA, Kaizen, and predictions) in the UI as visible locked modules ("coming in Phase 2 of your roadmap"). A locked module with a clear value-proposition sells; a half-built module destroys trust.

3. Personalized Dashboards & Trust State

Dashboard design is governed by a strategy of composition over duplication. Instead of building isolated visual pipelines, a unified widget library is filtered and composed per user role.

Widget Architecture

Every widget is a standardized module combining three pillars:

  • Event Queries: Executed deterministically over canonical events. No metric values may come from LLM outputs.
  • Visual Representation: Clean HTML/SVG metrics and charts displaying responsive, high-contrast layouts.
  • Provenance Hooks: Clicking a metric triggers an interactive visual trace (the "beam") directly highlighting its calculation lineage (raw Excel cells, input logs, or validation rule engines).

Trust Badges: Global UI Law

Under no circumstance may a numerical value display on a dashboard without a corresponding trust state badge. This satisfies PRD §22 regulatory audit compliance:

VERIFIED

The raw data points map directly to Excel cells or digital entry audits with zero mismatch.

RULE-ASSUMED

Calculated using logic rules (e.g. stage-wise scrap weight extrapolated via standard density constants).

UNRESOLVED

The data displays a mathematical warning or unresolved cell mapping that requires manual review.

4. System Architecture Directives

Data integrity must remain consistent across ingestion pathways. The append-only Canonical Event Store acts as the central spine. Roles modify their view representations but never fragment the underlying events.

Ingestion Flow & Data Collections

MO!D accommodates excel sheets and manual forms concurrently. Shop floors are not forced to abandon their existing spreadsheet workflows.

Dynamic Form Generation

Rather than hardcoding manual shift entry inputs, shop-floor forms are dynamically rendered from the learned process ontology (e.g., fields adjust automatically based on granulation or tableting stage attributes).

AuthN & RBAC Mapping

Supabase Auth handles security. Role boundaries (Steward, QE, GM) are resolved at the API layer against granular permissions (e.g. findings:gm-authority) to ensure data compliance.

Why Financial-Lite is in V1
Pharma GMs and Executives think in terms of commercial losses (rupees), not just rejection percentages. By implementing a single config mapping of ₹ loss per rejected unit per stage, MO!D transforms standard technical charts into high-impact financial dashboards.

5. UI Design Directives

The application interface relies heavily on high typographic hierarchy, density of information, and responsive layout elements.

App Shell Architecture

A persistent left-navigation rail structure provides workspace switching (Dashboard, Diagnostics, Data, Reports, Admin). The top header hosts a global scope bar (Batch, Product, Period) that dynamically filters every workspace widget.

Print-First Paradigm

Every analytical view features a standard action button to export or print. The Plant Director's monthly reporting module is formatted specifically to fit the plant's existing executive reporting template.

Additional UI Directives
Language Options

Supports English UI chrome with a configurable toggle for Tamil labels in data entry and findings cards to assist floor staff.

Verify-View Beams

When verifying direct entry inputs instead of imported spreadsheets, the beam line terminates at the user entry record and timestamps.

Educational Empty States

First-run screens must clearly outline what data will appear and what it represents. A new login must teach itself without documentation.

7. Open Questions for Client Consultation

Do not make assumptions regarding these parameters. Clarify directly with plant management during the kick-off review:

  1. What is the estimated financial loss value (₹ cost) per rejected unit at each process stage (Granulation, Tableting, Coating)?
  2. Is an initial cloud-hosted deployment pilot acceptable, or does regulatory data compliance mandate an on-premise infrastructure from day one?
  3. Are Tamil UI text labels required for shop-floor screens, or is English UI with standard data symbols sufficient for operators?
  4. Who are the named team members assigned to each user role for training and licensing allocations in the pilot phase?
