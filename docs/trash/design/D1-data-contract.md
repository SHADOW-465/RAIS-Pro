# D1 — Canonical Event & Data Contract

**Status:** **FROZEN v1.0.0** (2026-06-11). Self-reviewed against the decision-gate checklist: every §2 evidence row has a contract consequence; `d1-contract.ts` typechecks under `--strict`; every observational event type has ≥1 real example from `DATA/` (`Correction`/`Annotation` are system-generated and have none by design); §9 open questions are all non-blocking with documented v1 defaults. Breaking changes hereafter require a version bump + migration note here.
**Companion artifacts:** [`d1-contract.ts`](d1-contract.ts) (Zod definitions), `DATA/profile_d1_output.md` (raw structural evidence, generated by `DATA/profile_for_d1.py`)
**Downstream consumers:** D2 (validation rule catalog), D3 (findings/rulebook), D4 (pipeline architecture)

---

## 1. Purpose & scope

This contract defines the **atomic event model** every RAIS/MO!D module reads and writes. It is the boundary between the messy world (client Excel files) and the deterministic world (validation, analytics, reporting).

Core stance, inherited from the verification-loop design:

> Never fix data, never silently accept it. Ingestion is **read-only observation**: we record what the sheet *says*, with provenance and confidence. Disagreements between what the sheet says and what arithmetic implies become **Findings** (D3), never silent corrections.

Scope of v1: the six workbook families in `DATA/` (assembly rejection, balloon & valve integrity, shopfloor rejection, visual inspection, two cumulative summaries) for one client (Disposafe). The contract must extend to future MO!D clients without breaking changes, so client-specific vocabulary (stages, defect codes) lives in **registries**, not in the event schema.

## 3. Event model

### 3.1 Common envelope (all events)

Every event is an immutable record with this envelope:

| Field | Type | Req | Notes |
|-------|------|-----|-------|
| `eventId` | string | ✓ | Content hash of (type, payload, provenance) — stable across re-ingestion of identical data |
| `eventType` | enum | ✓ | One of §3.2 |
| `schemaVersion` | string | ✓ | This contract's semver, `"1.0.0"` at freeze |
| `ingestionId` | string | ✓ | Groups all events from one upload run |
| `occurredOn` | Period | ✓ | The *business* time (§5.3) — day, week, month, or fiscal year |
| `provenance` | Provenance | ✓ | §5.2. Where in which file this was read from |
| `confidence` | Confidence | ✓ | §5.5. How sure extraction is that the value means what we say it means |
| `extractedBy` | string | ✓ | `"heuristic"` or model id (e.g. `"llm:claude-fable-5"`) |
| `recordedAt` | ISO datetime | ✓ | Wall-clock ingestion time |
| `supersededBy` | string | — | Set only by a later `Correction` event (§6) |

### 3.2 Event types

**`Production`** — a quantity entered a stage for processing.
Payload: `stageId`, `quantity` (int ≥ 0), optional `batchNo`, `size`, `unit` (default `"pcs"`; SHOPFLOOR uses `"trolleys"`).
Examples: ASSEMBLY `VISUAL QTY`, VISUAL `REC. QTY`, SHOPFLOOR `No of TROLLEYS`.

**`Inspection`** — a disposition outcome at a stage: how many of the checked quantity went where.
Payload: `stageId`, `disposition` (`accepted | rejected | hold | downgrade`), `quantity`, optional `batchNo`, `size`.
`downgrade` covers VISUAL's `LOCAL GRADE` / `A GRADE` split. One sheet row typically yields one `Production` + 2–4 `Inspection` events.

**`Rejection`** — rejected quantity attributed to a defect category. The Pareto atom.
Payload: `stageId`, `defectCode` (registry id, nullable when unresolvable), `defectCodeRaw` (verbatim header/label), `quantity`, optional `batchNo`, `size`.
A day's rejections are *both* an `Inspection(disposition=rejected)` total *and* per-defect `Rejection` events; the validation engine checks they reconcile (D2), the contract does not assume they do (evidence row 5: they don't always).

**`Carryover`** — quantity crossing a period or stage boundary outside normal flow: hold resolutions, week blocks spanning months, reconciliation of stage N accepted → stage N+1 checked.
Payload: `fromRef` / `toRef` (period or stageId), `quantity`, `kind` (`hold-resolution | period-bridge | stage-handoff`).
v1 emits these sparsely (only where the sheet makes the link explicit, e.g. the JULY→AUGUST formulas); most carryover is *detected* by D2, then confirmed via adjudication.

**`AggregateClaim`** — a value the sheet *computed or asserted* rather than recorded: weekly/monthly/yearly totals, percentage cells, cross-sheet and cross-file pulls.
Payload: `claimKind` (`sum | percentage | external-pull | derived`), `statedValue` (number | `"#DIV/0!"` | other error string), `formulaText` (nullable — cached-only externals have none we can resolve), `aboutStageId?`, `aboutDefectCode?`, `aggregation` (`daily | weekly | monthly | fiscal-year`).
**Hard rule: analytics never read `AggregateClaim.statedValue` as input.** Claims exist so D2 can recompute and compare — they are the richest source of Findings (evidence rows 4–5).

**`Correction`** — supersedes a prior event (append-only fix path, §6).
Payload: `supersedesEventId`, `replacementEventId?`, `reason`, `authorisedBy` (adjudication ref).

**`Annotation`** — human or system note attached to events/cells; the output shape of adjudication (D3 will extend, not replace, this type).
Payload: `targetEventIds[]` and/or `targetCells[]`, `text`, `author` (`steward | gm | system`), `findingId?`, `verdict?` (`mistake | intentional | unsure`).

**`Dispatch`** — quantity shipped (appears only in CUMULATIVE, sourced externally).
Payload: `quantity`, `unit`. Always carries `externalRef` provenance and capped confidence (§5.5). Kept as its own type because the GM's report needs it, but it is *unverifiable* in v1 — its lineage state can never exceed "assumed" until the client provides the dispatch workbook.

> Plan's `ProductionEvent / InspectionEvent / RejectionEvent / CarryoverEvent / CorrectionEvent / AnnotationEvent` list is preserved; `AggregateClaim` and `Dispatch` were added because the real files demanded them (evidence rows 4, 10).

## 5. Field semantics

### 5.1 Quantities
Non-negative integers in v1 (every observed quantity cell is a whole count). Percentages are **never** stored on observation events — they are always `AggregateClaim`s, because every percentage in the files is derived (and sometimes derived wrongly, evidence row 5). Unit defaults to `pcs`; `trolleys` for shopfloor. Unit mismatches across stages are a D2 range/unit check, not a contract concern.

### 5.2 Provenance

```
Provenance = {
  file: string,           // e.g. "ASSEMBLY REJECTION REPORT.xlsx"
  fileHash: string,       // sha256 of the uploaded bytes
  sheet: string,          // verbatim, including " MAY 25" leading-space quirks
  tableId: string,        // "t1" default; "t2" for second block on a sheet (evidence row 2)
  cells: string[],        // A1-style refs this event was read from, ≥1
  headerPath: string[],   // verbatim header text(s) above the cell, top row first
  rowLabel: string | null,// verbatim col-A label (date string, batch no, "Total", …)
  formulaText: string | null,  // raw formula if the cell had one
  cachedValue: string | number | null, // Excel's stored result, if it had a formula
  externalRef: string | null   // e.g. "[2]MAY25 PRIMARY OUTPUT!$E$36" (evidence row 10)
}
```

Provenance is sufficient to power the verify-mode beam (KPI → cell) and the Finding cards' "show me the cells" panel without re-opening the workbook with an LLM.

### 5.3 Dates and periods

Canonical `Period`: `{ kind: "day" | "week" | "month" | "fiscal-year", start: ISOdate, end: ISOdate }`.

Normalization rules (deterministic, no LLM):
1. Datetime cells → `day`.
2. Integer in a date-typed column position, plausible Excel serial (40000–50000) → serial→date (1900 system) → `day`. (SHOPFLOOR col A, VISUAL `Effective Date` cells.)
3. Sheet-name months (`APRIL 25`, `JUNE 2025`, ` MAY 25`) → `month`, trimmed/case-folded; the **fiscal year context** comes from the workbook (`2025-26` = 2025-04-01 → 2026-03-31).
4. Weeks are *only* created from explicit week blocks (rows between two weekly-total rows, or `WEEK n` labels); week periods may cross months (evidence row 7) and must store real start/end dates.
5. A `day` event dated outside its host sheet's month is **kept as written** + flagged (D2 check), not coerced.

### 5.4 Row classification (extraction contract)

Extraction must classify every non-empty row as exactly one of: `data`, `subtotal-weekly`, `total-monthly`, `percentage`, `marker` (SUNDAY/HOLIDAY/W. REPORT labels), `legend`, `header`, `doc-meta` (Doc. No / Supersedes / Effective Date block), `unknown`. `data` rows yield observation events; `subtotal/total/percentage` rows yield `AggregateClaim`s; `marker`/`legend`/`doc-meta` yield nothing (but markers inform calendar checks in D2 — a SUNDAY marker is evidence the plant was idle); `unknown` rows (e.g. VISUAL `APRIL 25` R37's bare float row) yield a Finding candidate. **No row is silently dropped.**

### 5.5 Confidence

```
Confidence = {
  score: number,        // 0..1
  basis: "exact" | "heuristic" | "llm" | "external-cached"
}
```

- `exact` (1.0): value read from a typed cell whose header resolved to a registry entry by exact alias.
- `heuristic` (≤0.9): positional/format inference (e.g. serial-date detection).
- `llm` (≤0.85): semantic classification by the model (column meaning, table boundaries).
- `external-cached` (≤0.5 **cap**): values whose formula points outside the uploaded file set (evidence row 10). Cannot reach "verified" lineage state without the source file.

Confidence is about *extraction fidelity* ("did we read what they wrote correctly?"). Whether what they wrote is *true* is the validation engine's domain — kept strictly separate.

## 6. Immutability & versioning

1. **Append-only.** No event is ever updated or deleted. Re-ingesting an identical file produces identical `eventId`s (content-hash) → natural dedupe; a changed file produces new events under a new `ingestionId` + `fileHash`.
2. **Corrections supersede.** The only fix path is a `Correction` event setting `supersededBy` on the old event. Analytics read the *effective* set (events minus superseded). Corrections require an adjudication reference — the system never auto-corrects.
3. **Adjudications are annotations.** Steward/GM verdicts are `Annotation` events; they change an event's *lineage state* (verified / rule-based-assumption / unresolved), never its payload.
4. **Schema versioning.** `schemaVersion` is semver. Additive optional fields = minor bump, readers ignore unknown fields. Any breaking change = major bump + migration note in this doc. Registries version independently (`registryVersion` recorded on each ingestion run).
5. **The original file is the ultimate provenance.** Uploaded bytes are retained content-addressed (`fileHash`); every event must remain re-derivable from file + extraction code version + registry version.

## 7. LLM-facing subset (cross-provider constraints)

Only the *candidate event* shapes produced during ingestion are LLM-facing (semantic classification of tables/columns/rows). Those Zod schemas follow the existing rules from `src/lib/schemas.ts`: `.nullable()` instead of `.optional()`, plain ints (no literal unions), strings for value fields. Everything else in this contract (stored events, registries, claims) is internal and uses idiomatic Zod freely. The split is explicit in `d1-contract.ts` (`Candidate*` schemas vs canonical schemas). **The LLM never emits quantities that get stored** — it emits *cell references and role labels*; deterministic code reads the values out of those cells (pipeline invariant #1: the model never does maths, and it also never transcribes numbers).

## 8. Worst-sheet acceptance tests (freeze checklist)

The contract is frozen only when extraction + this schema can represent every case below without loss or coercion. Each is a concrete cell in `DATA/`:

1. **ASSEMBLY NOV 25, R5/E** — header `REJ %` living in the *second* header row of a merged two-row header → `headerPath` captures both rows.
2. **ASSEMBLY APRIL 25, R29** — `#DIV/0!` on a zero-production day → `AggregateClaim.statedValue = "#DIV/0!"`, no observation events for the empty quantities.
3. **ASSEMBLY AUG 25, R15/B** — string `HOLIDAY` inside a numeric column → row classified `marker`, zero events, no crash.
4. **BALLOON APRIL 25** — two tables, one sheet → distinct `tableId`s; defect summary block (R54–59) referencing `=S37` etc. → `AggregateClaim(claimKind="derived")` with `formulaText`.
5. **SHOPFLOOR APRIL 25, A4** — serial date `45748` → `Period{day, 2025-04-01}` at `heuristic` confidence.
6. **SHOPFLOOR APRIL 25, K30** — grand total formula omitting col I → claim stored verbatim; recomputation lives in D2 (this exact cell is the demo Finding).
7. **CUMULATIVE Sheet1, C4** — `='[1]APRIL 25'!$N$32` cached `247767`-class values → `externalRef` set, confidence `external-cached ≤ 0.5`.
8. **VISUAL 4-2-25** — batch-grain rows (`25A28`, size `16FR`), date only in `B4` as serial → `batchNo`/`size` populated; `occurredOn` from the sheet-level date, single-day sheet named by date.
9. **VISUAL JULY 25 weekly totals** — formula spanning `'AUGUST 25'` → week `Period` with real cross-month start/end; claim keeps full `formulaText`.
10. **VISUAL APRIL 25, R34** — stated REJ 1708 vs reason-matrix sum 1544 → both representable independently (one `Inspection(rejected)`, many `Rejection`s); no forced reconciliation.
11. **VISUAL APRIL 25, R37** — unlabeled row of floats → `unknown` row class → Finding candidate, not dropped.
12. **ASSEMBLY YEARLY, Q19** — `=E19+I19+M19` summing percentages → claim captured; D2's recomputation will flag the method, demonstrating we catch *their* report's logic errors.
13. **VISUAL FORMATE sheet** — blank template → zero events, sheet recorded in ingestion manifest as `skipped: template`.

## 9. Open questions (all non-blocking; v1 defaults in force until answered)

1. Is the VISUAL file's inspection the same station as ASSEMBLY's VISUAL column? (PA — affects stage registry & conservation checks.) **v1 default: same station**; D2 conservation checks across the two files run at `info` severity until confirmed.
2. `HOLD QTY` downstream fate — re-inspected into which stream? (PA.) **v1 default:** hold resolutions are not synthesized; D2 only *detects* unbalanced holds and emits Findings.
3. Do we ask the client for the dispatch/"primary output" workbooks? (GM-report implications, D6.) **v1 default:** `Dispatch` stays `external-cached`, lineage capped at "assumed"; D6 renders its trust mark accordingly.
4. Trolley→pieces conversion factor for shopfloor, if any. **v1 default:** no conversion; shopfloor conservation checks stay within `trolleys`, cross-unit checks disabled.
